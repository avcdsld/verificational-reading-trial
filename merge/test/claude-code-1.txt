### Merge.sol の構造

Pak の NFT アートプロジェクト「merge.」のコントラクト。ERC721 を独自実装しており、以下の特殊な仕組みがあります:

**核心メカニズム — マージ(合体)**
- 各トークンは `class`(1〜4）と `mass`（1〜99,999,998）を持つ（`value = class * 100M + mass` でエンコード）
- 非ホワイトリストのアドレスは **トークンを1つしか持てない**
- 非ホワイトリストアドレスにトークンを転送すると、既存トークンと自動的に **マージ** される（mass が大きい方が生き残り、小さい方がバーンされる）
- 最大 mass を持つトークンが「Alpha」になる

**アクセス制御**
- `onlyPak`: 管理者操作（freeze/thaw, finalize, ホワイトリスト管理など）
- `onlyValidSender`: `INiftyRegistry` 経由でミント権限チェック
- `onlyValidWhitelist`: ホワイトリストアドレスのみ（`merge()` 関数用）
- `notFrozen`: freeze 中は転送・バーン不可

**主要関数**
- `mint()` — バッチミント（`onlyValidSender`）
- `merge()` — 明示的マージ（`onlyValidWhitelist`）
- `_transfer()` — 転送＋自動マージロジック
- `burn()` — トークンバーン
- `freeze()` / `thaw()` / `finalize()` — ステート管理
- `encodeClassAndMass()` / `decodeClassAndMass()` — エンコード/デコード

**依存コントラクト**
- `INiftyRegistry` — 外部レジストリ（テストではモック必要）
- `IMergeMetadata` / `MergeMetadata` — メタデータ生成

## 確認結果

### Q1. Class の目的と固定性
- **目的**: ビジュアルの色テーマ（1=白/黒, 2=金/黒, 3=白/青, 4=白/赤）。値は `class × 1億 + mass` でエンコード
- **固定**: マージ時は `_values[tokenIdLarge] += massSmall` で mass 部分だけ加算。総mass < 1億 が保証されているのでclass桁に繰り上がらない → **classは変わらない**。死んだ側のclassは消失

### Q2. トークン発行数
- 上限数の定義はない。制約は `_massTotal < 99,999,999`（全トークンのmass合計）と `finalize()` 後のミント禁止のみ
- sentinel値（`value % 1億 == 99,999,999`）はスキップされる（IDは消費するがトークンは生成されない）

### Q3. Mass の最大値が 99,999,998 の理由
- `99,999,999` は sentinel（ミント時のスキップフラグ）として予約されている
- エンコードで class 桁との衝突を防ぐ構造上の制約

### Q4. マージ時の burn イベント
- **出る**: 明示的 `merge()` → `Transfer(owner, address(0), deadToken)`、転送時の自動マージ → `Transfer(to, address(0), deadToken)`。いずれも `address(0)` への Transfer で表現（ERC721 標準のバーン表現）

### Q5. Alpha とは
- **システム内で最大 mass を持つトークン**。`_alphaId` と `_alphaMass` で追跡
- ミント時・マージ時に `combinedMass > _alphaMass`（厳密な `>`）で更新。メタデータでは色が反転する特別な表示

### Q6. NiftyRegistry の権限
- `mint()` と `batchSetMergeCountFromSnapshot()` のアクセス制御に使用（`onlyValidSender` modifier）
- `_registry` は `immutable` でコンストラクタ後に変更不可

### Q7. メタデータ
- オンチェーン SVG 生成。mass に応じた半径の円を描画。`data:application/json;base64,...` 形式で返す
- 属性: Mass, Alpha(0/1), Tier(class), Class(tokenId由来), Merges(マージ回数)

### Q8. 同じ mass のマージ
- `_merge(tokenIdRcvr, tokenIdSndr)` で `massRcvr >= massSndr` の場合 Rcvr が生存 → **等しいときは Receiver（第1引数）が残る**
- 転送時の自動マージでは既存トークンが Receiver → **既存トークンが残る**

### Q9. 独自イベント
- `AlphaMassUpdate(uint256 indexed tokenId, uint256 alphaMass)` — Alpha 変更時
- `MassUpdate(uint256 indexed tokenIdBurned, uint256 indexed tokenIdPersist, uint256 mass)` — マージ時（combined mass）とバーン時（0, 0）

## コードから読み取れる証拠

**1. `batchSetMergeCountFromSnapshot`** — 関数名に「Snapshot」とある

```solidity
// Merge.sol:585-591
function batchSetMergeCountFromSnapshot(uint256[] calldata tokenIds_, uint256[] calldata mergeCounts_) external onlyValidSender {
    require(!_mintingFinalized, "Merge: Minting is finalized.");
    ...
}
```

マージ回数を外部から直接書き込める関数です。つまり、**このコントラクトの外で既にマージが行われていた**状態を復元する目的です。

**2. `_omnibus`（一括管理アドレス）** — Nifty Gateway の特徴的なパターン

```solidity
// Merge.sol:541
_owners[index] = omnibus;  // 全トークンがまず omnibus に発行される
```

Nifty Gateway はユーザーの NFT をプラットフォーム側のアドレス（omnibus）で一括管理するカストディアル方式です。

**3. `INiftyRegistry`** — Nifty Gateway のアクセス制御

```solidity
// Merge.sol:42-44
interface INiftyRegistry {
   function isValidNiftySender(address sending_key) external view returns (bool);
}
```

ミント権限を Nifty Gateway のレジストリに委ねています。

**4. `finalize()` と `freeze()`/`thaw()`**

```solidity
// Merge.sol:593-605
function finalize() external onlyPak {
    thaw();        
    _mintingFinalized = true;        
}

function freeze() external onlyPak {
    require(!_mintingFinalized);
    frozen = true;
}
```

`freeze` は finalize 前のみ使えます。移行作業中にトークン移動を一時停止し、データ投入が完了したら `finalize` で確定、という流れです。

## 推測される移行フロー

```
1. コントラクトデプロイ（freeze 状態で開始可能）
2. mint() で既存トークンを再現（sentinel で欠番を維持）
3. batchSetMergeCountFromSnapshot() で過去のマージ履歴を復元
4. finalize() で移行を確定（以後ミント不可、thaw で移動解禁）
```

sentinel はこの手順2で、元のシステムで既に消滅していたトークンIDの「穴」を正確に再現するために必要だった、ということです。

## sentinel の実際の用途

オリジナルのコントラクトに**バグがあった**ことが判明しています（同じ NFT が同時に2つのアドレスに所有されうる問題）。そのため約7日後に新コントラクトがデプロイされました。

この文脈で sentinel の役割が明確になります:

1. オリジナルのコントラクトで売却・マージが進行済み（312,686 units、28,983人の購入者）
2. 新コントラクトに**既存状態を再現**する必要があった
3. オリジナルで既にマージや消滅したトークンIDを**欠番にしつつ**、生きているトークンのIDを維持するために sentinel を使用
4. `batchSetMergeCountFromSnapshot()` で過去のマージ回数も復元

つまり先ほどの議論での推測（コントラクトのアップグレード時の状態移行）が正解だったということですね。sentinel はバグ修正版への移行で、元のトークンID体系を壊さずに再ミントするための仕組みでした。

> これは興味深い。これ、リリース後に移行されたということですかね」

はい、そうです。流れとしては:

1. **2021年12月** — Nifty Gateway で Merge セール開始、オリジナルのコントラクト（`0x27d2...`）にデプロイ
2. **セール中/直後** — バグ発覚（同じ NFT が2つのアドレスに同時所有される問題）
3. **約7日後** — 修正版コントラクト（`0xc3f8...`、このリポジトリのコード）をデプロイし、状態を移行

このコードに移行用の機能が揃っているのはそのためです:

- `mint()` + sentinel → 生存トークンだけを元のIDで再ミント、消滅済みIDは欠番
- `batchSetMergeCountFromSnapshot()` → 旧コントラクトでのマージ回数を復元
- `freeze()` / `thaw()` → 移行作業中にトークン移動を一時停止
- `finalize()` → 移行完了後にミントを永久に封鎖

これらの機能が `!_mintingFinalized` でガードされているのも、移行完了後は二度と使わないという前提だからですね。
